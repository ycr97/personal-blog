# 电商订单智能审单引擎的设计与实现

> 原文链接：[语雀：电商订单智能审单引擎的设计与实现](https://www.yuque.com/u12614739/mcmytl/rfc6zy95mk417zu8)

> 本文记录了一次电商订单自动审核引擎的设计过程，从业务问题出发，逐步推导出一套基于规则引擎 + 策略模式 + 延迟消息的可扩展审单架构。

## 一、业务背景与问题分析

在电商履约场景中，订单从平台同步到 ERP 后，需要经过审核才能下发到仓库进行拣货发货。传统做法是人工审核，但随着订单量增长，这种方式暴露了几个核心问题：

- 审单规则会持续变化 —— 不能把规则硬编码在业务流程里，否则每次调整都要改代码上线

- 审单不是简单的二元判断 —— 除了"通过/拒绝"，还有"当前时点不适合判断"的场景（如刚支付的订单可能很快退款）

- 审单过程必须可追溯 —— 线上出了问题需要能解释"为什么这单被拦了"或"为什么这单放行了"

- 审单不能强耦合订单下发 —— 否则规则越来越复杂时，主链路会越来越脆

从这些问题出发，我的设计目标是：把"规则判断"从散落的 if-else 升级成"配置驱动的审单引擎"。

## 二、整体架构设计

### 2.1 三层架构

我将整个审单流程拆解为三个层次，每层职责单一、边界清晰：

```text
┌───────────────────────────────────────────────────────────┐
│                     AuditEngine（引擎层）                    │
│              负责编排流程、决策路由、日志记录                    │
├───────────────────────────────────────────────────────────┤
│                     RuleExecutor（执行层）                   │
│             负责加载验证器、按 AND 逻辑组合执行                 │
├────────────┬────────────┬────────────┬────────────────────┤
│   Scope    │  Product   │    Tag     │   PaymentTime      │
│  Validator │  Validator │  Validator │   Validator         │
│  (范围校验) │ (商品校验)  │ (标签校验)  │  (支付时间校验)      │
└────────────┴────────────┴────────────┴────────────────────┘
```

- 引擎层（AuditEngine）：流程编排者，不关心具体校验逻辑，只负责"加载规则 → 执行匹配 → 路由结果"

- 执行层（RuleExecutor）：持有所有验证器的注册表，负责按配置逐个调用验证器并执行 AND 组合

- 验证层（RuleValidator 实现）：每个验证器只关注自己领域的判断逻辑

### 2.2 审核结果三态设计

审单结果不是简单的"通过/不通过"，而是三种状态：

| 状态 | 枚举值  | 含义                               | 后续动作                               |
| ---- | ------- | ---------------------------------- | -------------------------------------- |
| 通过 | PASS    | 未命中任何拦截规则，且支付时间充足 | 自动下发订单                           |
| 拒绝 | REJECT  | 命中了拦截规则                     | 进入人工审核队列                       |
| 待定 | PENDING | 未命中拦截规则，但支付时间不足     | 发送 RocketMQ 延迟消息，到期后自动重审 |

PENDING 状态是一个关键设计决策。它源于一个真实的业务痛点：刚支付的订单可能很快退款，如果立即审核通过并下发到仓库，后续退款会导致已拣货的订单需要拉回，造成人力和物流浪费。通过引入一个"冷静期"，在支付后等待一段时间再决定是否下发，可以显著降低这类损失。

### 2.3 领域模型收敛

引擎不直接依赖 EcOrderHeadDO、EcOrderLineDO 这类重业务实体，而是先通过 OrderRuleValidateConverter 转换成轻量的 OrderRuleValidateDTO，只保留审单所需的最小上下文：

```text
OrderRuleValidateDTO
├── orderId        // 平台订单号
├── shopCode       // 店铺编码
├── channelCode    // 渠道编码
├── orderType      // 订单类型
├── items          // 商品列表 (sku + productName + quantity)
├── sellerFlag     // 卖家旗标 (0-7)
├── tags           // 订单标签
└── paymentTime    // 支付时间
```

设计意图：让引擎只处理审单所需的最小上下文，而不是理解整个订单系统。订单领域对象怎么变，不应该直接影响引擎内部模型。

## 三、核心流程

### 3.1 审单主流程

```text
订单进入审核
    │
    ▼
① 参数校验（空订单直接 PASS）
    │
    ▼
② 加载规则：按 shopCode + channelCode 查询启用的规则，按 priority DESC 排序
    │
    ▼
③ 遍历规则，逐条匹配：
    │
    ├── 命中 ──→ handleReject()：异步记录日志 → 返回 REJECT
    │
    └── 全部未命中 ──→ 继续 ④
    │
    ▼
④ 检查延迟审核：遍历规则找到第一个配置了 delayMinutes 的规则
    │
    ├── 存在延迟配置，且支付时间不足 ──→ handleDelay()：
    │       同步写日志(获取logId) → 发送 RocketMQ 延迟消息 → 返回 PENDING
    │
    └── 无延迟配置 或 支付时间已充足 ──→ 返回 PASS
```

### 3.2 延迟重审流程

```text
RocketMQ 延迟消息到期，消费者收到消息
    │
    ▼
① 检查重试次数（最多 3 次）
    │
    ▼
② 更新审核日志状态为 EXECUTED
    │
    ▼
③ 重新验证支付时间
    │
    ├── 通过 → 更新订单状态为 PASS → 调用订单创建服务下发
    │
    └── 失败 → 标记 FAILED（理论上不应发生，延迟时间已足够）
```

延迟重审只验证支付时间这一个变量，而不是重新执行整套规则。因为延迟的原因只有一个——支付时间窗口未成熟，那么重审时只需要验证这个条件是否满足。这是一种务实的"最小重放"设计。

## 四、设计模式详解

### 4.1 策略模式（Strategy Pattern）—— 验证器的可插拔设计

这是整个设计中最核心的模式。定义统一的验证器接口：

```text
public interface RuleValidator {
    /**
     * @param order     订单DTO
     * @param configStr 该验证器的 JSON 配置
     * @return true-通过/符合, false-不符合
     */
    boolean validate(OrderRuleValidateDTO order, String configStr);
}
```

每个具体验证器通过 @Component(value = "xxx_validator") 注册到 Spring 容器。RuleExecutor 利用 Spring 的 Map<String, RuleValidator> 自动注入机制，将所有实现类以 Bean 名称为 Key 收集到一个 Map 中：

```text
@Component
public class RuleExecutor {
    @Resource
    private Map<String, RuleValidator> validatorMap;  // Spring 自动收集所有 RuleValidator 实现

    @PostConstruct
    public void init() {
        log.info("规则执行器初始化完成，已注册验证器数量：{}", validatorMap.size());
    }
}
```

当前实现了四个策略：

| 验证器               | Bean Name              | 职责                    | 配置模型                                                        |
| -------------------- | ---------------------- | ----------------------- | --------------------------------------------------------------- |
| ScopeValidator       | scope_validator        | 校验店铺/订单类型白名单 | ScopeConfig{shopCodes, orderTypes}                              |
| ProductValidator     | product_validator      | 商品 SKU 黑名单/白名单  | ProductConfig{mode, skus}                                       |
| TagValidator         | tag_validator          | 卖家旗标和订单标签过滤  | TagConfig{excludeFlags, includeFlags, excludeTags, includeTags} |
| PaymentTimeValidator | payment_time_validator | 支付后时间间隔校验      | 直接参数传入                                                    |

为什么用 Spring Map 注入而不是手动注册？ 利用框架能力，新增验证器只需要实现接口 + 加注解，零修改引擎和执行器代码。这是开闭原则（OCP）的典型应用。

### 4.2 规则组合 —— AND 逻辑的多验证器编排

一条规则（EcAuditRuleDO）的 configJson 是一个 JSON 数组，每个元素对应一个验证器调用：

```text
[
  {"validateCode": "scope_validator",   "validateName": "范围验证",
   "paramsSchema": "{\"shopCodes\":[\"SHOP001\"],\"orderTypes\":[\"normal\"]}"},
  {"validateCode": "product_validator", "validateName": "商品验证",
   "paramsSchema": "{\"mode\":\"BLACK\",\"skus\":[\"SKU001\",\"SKU002\"]}"},
  {"validateCode": "tag_validator",     "validateName": "标签验证",
   "paramsSchema": "{\"excludeFlags\":[1,2]}"}
]
```

RuleExecutor.match() 对数组内所有验证器执行 AND 逻辑——全部通过才算匹配成功：

```text
public boolean match(OrderRuleValidateDTO order, List<RuleConfig> configList) {
    for (RuleConfig config : configList) {
        RuleValidator ruleValidator = validatorMap.get(config.getValidateCode());
        if (!ruleValidator.validate(order, config.getParamsSchema())) {
            return false;  // 任一验证器不通过 → 整条规则不匹配
        }
    }
    return true;  // 全部通过 → 规则匹配（即命中拦截）
}
```

而多条规则之间是 OR + 优先级 的关系——按优先级从高到低遍历，第一条匹配即返回 REJECT。

这种设计让规则具有很强的表达能力：

- 单维度拦截：一条规则只配一个验证器，如"拦截所有包含 SKU001 的订单"

- 多维度交叉拦截：一条规则配多个验证器，如"拦截 SHOP001 店铺中包含黑名单商品且带红旗标记的订单"

### 4.3 责任链变体 —— 规则优先级遍历

多条规则的遍历采用了责任链的思想，但做了简化：

```text
规则1 (priority=100) ──匹配──→ REJECT（短路返回）
        │ 不匹配
        ▼
规则2 (priority=90)  ──匹配──→ REJECT（短路返回）
        │ 不匹配
        ▼
规则3 (priority=80)  ──匹配──→ REJECT（短路返回）
        │ 不匹配
        ▼
   全部未命中 ──→ 检查延迟审核 ──→ PASS 或 PENDING
```

与经典责任链的区别：这里不需要每个节点决定是否传递给下一个，而是由引擎统一遍历，命中即返回。这种简化降低了复杂度，同时保留了优先级排序的灵活性。

此外，在"命中"之前还有一层粗筛：规则先按 shopCode + channelCode 从数据库过滤，避免无效的全量扫描。这体现了"先粗筛，再精筛"的思路。

### 4.4 异步非阻塞 —— 日志不拖累主流程

审核日志的写入采用 @Async 异步执行：

```text
@Async("taskExecutor")
public void insertAuditLogAsync(EcAuditLogDO auditLog) {
    try {
        logMapper.insert(auditLog);
    } catch (Exception e) {
        log.error("审核日志异步入库异常", e);
        // 吞掉异常，不影响审核主流程
    }
}
```

设计考量：审核日志是"观测数据"，不是"业务数据"。日志写入失败不应该阻断订单的审核流程。

唯一的例外：handleDelay 中的日志插入是同步的，因为需要拿到日志 ID 放入延迟消息，后续消费时通过这个 ID 更新日志状态。不同数据有不同的一致性要求，不应该一刀切。

### 4.5 延迟消息模式 —— 基于 RocketMQ 的定时重审

这是整个设计中最有业务价值的部分。

DelayAuditMessage 的设计体现了"消息自包含"原则——消息体携带了重审所需的全部信息：

```text
DelayAuditMessage
├── orderId / shopCode / channelCode / orderType  // 订单标识
├── paymentTime                                    // 支付时间（毫秒时间戳）
├── delayMinutes / strictMode                      // 延迟配置
├── ruleId / ruleName                              // 规则关联
├── auditLogId                                     // 日志关联（用于状态回写）
├── retryCount / maxRetryCount(=3)                 // 重试控制
└── createTime                                     // 消息创建时间
```

消费者不需要额外查询数据库就能完成重审判断，降低了对数据库的依赖。

### 4.6 规则快照 —— 审计溯源的保障

每次审核都会将当时的规则配置 JSON 完整保存到 ec_audit_log.snapshot_json 字段：

```text
auditLog.setSnapshotJson(config);  // 保存规则配置快照
```

规则是可以随时修改的。如果某天需要追溯"这个订单当时为什么被拦截了"，只看当前规则可能已经变了。快照记录了审核那一刻的规则状态，是审计溯源的关键。

## 五、各验证器设计细节

### 5.1 ScopeValidator —— 范围白名单

最简单的验证器，校验订单的店铺和类型是否在白名单中：

- 如果配置了 shopCodes 且订单店铺不在其中 → 不匹配

- 如果配置了 orderTypes 且订单类型不在其中 → 不匹配

- 未配置则跳过（默认通过）

### 5.2 ProductValidator —— 商品黑白名单

支持两种模式：

- BLACK（黑名单）：订单中任一 SKU 在黑名单中 → 匹配（拦截）

- WHITE（白名单）：订单中所有 SKU 都必须在白名单中，否则 → 不匹配

实现上用 HashSet 做 SKU 集合比较，避免 O(n²) 遍历。

### 5.3 TagValidator —— 标签旗标校验

支持四个维度的组合过滤：

- excludeFlags：排除旗标列表，命中即失败

- includeFlags：指定旗标列表，不在其中即失败

- excludeTags：排除标签列表，有交集即失败

- includeTags：指定标签列表，无交集即失败

利用 Collections.disjoint() 做集合交集判断，代码简洁。

### 5.4 PaymentTimeValidator —— 支付时间校验

该验证器比较特殊：

- 不实现 RuleValidator 接口（因为返回值需要携带延迟时间等额外信息）

- 提供 validateWithResult() 方法，返回 ValidatorResult（包含 success、delayMinutes、elapsedMinutes）

- 支持严格模式（elapsed > delay 才通过）和非严格模式（elapsed >= delay 即通过）

```text
boolean shouldDelay = strictMode
    ? elapsedMinutes <= delayMinutes    // 严格：必须严格超过
    : elapsedMinutes < delayMinutes;    // 非严格：达到即可
```

## 六、容错设计

整个引擎在异常处理上遵循"宁可放行，不可阻塞"的原则：

| 异常场景                         | 处理策略                       | 原因                                 |
| -------------------------------- | ------------------------------ | ------------------------------------ |
| 规则配置 JSON 解析失败           | 跳过该规则，继续下一条         | 单条规则异常不应阻断整个审核         |
| 验证器执行异常                   | 返回 false（不匹配），继续流程 | 异常不等于"命中拦截"                 |
| 未找到 validateCode 对应的验证器 | 跳过该验证项，记录 warn        | 配置错误不应影响其他验证器           |
| 审核日志异步写入失败             | 吞掉异常，记录 error 日志      | 日志是观测数据，不是业务数据         |
| 延迟消息发送失败                 | 标记日志状态为 FAILED          | 降级为人工处理                       |
| 延迟消费者重试超过 3 次          | 停止重试，标记失败             | 防止消息风暴                         |
| 订单信息为空                     | 直接返回 PASS                  | 无信息无法判断，放行后由后续流程兜底 |

## 七、可扩展性验证

假设需要新增一个"订单金额验证器"，用于拦截超大额订单：

Step 1：定义配置模型

```text
@Data
public class AmountConfig {
    private BigDecimal maxAmount;
}
```

Step 2：实现验证器

```text
@Component(value = "amount_validator")
public class AmountValidator implements RuleValidator {
    @Override
    public boolean validate(OrderRuleValidateDTO order, String configStr) {
        AmountConfig config = JSONUtil.toBean(configStr, AmountConfig.class);
        return order.getTotalAmount().compareTo(config.getMaxAmount()) <= 0;
    }
}
```

Step 3：在管理后台配置规则时引用 amount_validator

总计修改：0 个已有文件。 新增 2 个文件（Config + Validator）。引擎、执行器、其他验证器完全不需要改动。

## 九、设计模式总结

| 设计模式            | 应用位置                                     | 解决的问题                   |
| ------------------- | -------------------------------------------- | ---------------------------- |
| 策略模式            | RuleValidator 接口 + 4 个实现                | 校验维度的可插拔扩展         |
| 注册表模式          | Spring Map<String, RuleValidator> 注入       | 验证器的自动发现和零配置注册 |
| 责任链变体          | AuditEngine 多规则优先级遍历 + 短路          | 多规则的有序匹配             |
| AND 组合            | RuleExecutor.match()                         | 单规则内多维度条件的组合     |
| 适配器/转换器       | OrderRuleValidateConverter                   | 隔离订单域与规则域           |
| 异步非阻塞          | AuditLogAsyncService + @Async                | 日志写入不阻塞主流程         |
| 事件驱动 / 延迟消息 | RocketMQ Producer / Consumer                 | 支付冷静期的定时重审         |
| 快照模式            | auditLog.snapshotJson                        | 规则变更后的审计溯源         |
| 消息自包含          | DelayAuditMessage 全量字段                   | 消费者无需额外查库即可重审   |
| 静态工厂方法        | ValidatorResult.success() / fail() / delay() | 简化结果对象的构造           |

## 十、思考与收获

- 从业务痛点推导设计，而不是为了用模式而用模式。三态结果（PASS / REJECT / PENDING）不是凭空设计的，而是分析"发货后退款"这个痛点后自然推导出来的。好的抽象来自对业务的深入理解。

- 利用框架能力简化设计。Spring 的 Map<String, Interface> 注入替代了手写的工厂/注册表，减少了样板代码，也消除了"忘记注册新实现"的风险。

- 区分"业务数据"和"观测数据"的一致性要求。审核日志用异步写入可以接受偶尔丢失，但延迟消息的日志 ID 必须同步获取。不同数据有不同的一致性要求，不应该一刀切。

- 可扩展性的终极检验是"新增一个维度需要改几个文件"。当前设计下答案是 0 个已有文件，这意味着扩展时不会引入回归风险。

- 把"不确定"转成"延后再判断"。很多时候业务不是非黑即白的，识别出"当前时点不适合做决策"并用工程化手段处理它，比强行同步判定要好得多。

## 十一、演进方向

- 统一 PaymentTimeValidator 的接口：让它也实现 RuleValidator，将延迟信息通过扩展返回值或上下文传递，保持验证器抽象的一致性。

- 死信队列与补偿机制：为延迟审核消息补齐死信处理，避免极端情况下消息丢失导致订单永远卡在 PENDING。

- 规则命中监控：给规则命中率、延迟审核触发率、最终通过率增加监控指标，形成运营闭环。

- 可视化规则配置：让运营人员在管理后台通过表单配置规则，而不是手写 JSON。

- 接入 LiteFlow 规则引擎：将自研的规则编排升级为 LiteFlow 驱动，获得更强的编排能力和规则热更新支持（详见下一章）。

## 十二、演进实践：改造为 LiteFlow 规则引擎

当前的自研引擎已经满足了基本的审单需求，但随着业务复杂度增长，存在一些天花板：

- 规则组合只支持 AND，不支持 OR、NOT、IF-ELIF-ELSE 等复杂逻辑

- 规则变更后需要重启或手动刷新，不支持热更新

- 多个验证器只能串行执行，无法并行加速

- 缺少执行链路的可视化追踪

LiteFlow 是一款轻量级的 Java 组件式规则引擎框架，天然支持上述能力。更关键的是，当前设计的验证器已经是单一职责、无状态的组件——这正是 LiteFlow 节点的理想形态，改造成本很低。

### 12.1 当前设计与 LiteFlow 的映射关系

| 当前设计                      | LiteFlow 概念          | 说明                             |
| ----------------------------- | ---------------------- | -------------------------------- |
| RuleValidator 实现类          | NodeComponent 节点     | 每个验证器变成一个 LiteFlow 节点 |
| RuleExecutor.match() AND 逻辑 | THEN(a, b, c) 串行编排 | 全部通过才算命中                 |
| 多规则优先级遍历 + 短路       | IF / SWITCH 条件编排   | LiteFlow 原生支持                |
| configJson JSON 数组          | EL 规则表达式          | 从 JSON 配置升级为 DSL 表达式    |
| OrderRuleValidateDTO          | 自定义上下文           | 放入 LiteFlow 的上下文中传递     |
| AuditResultEnum 三态          | 上下文中的结果变量     | 节点执行过程中写入上下文         |

### 12.5 规则表达式替代 JSON 配置

改造前：ec_audit_rule.config_json

```text

  {"validateCode": "product_validator", "paramsSchema": "{\"mode\":\"BLACK\",\"skus\":[\"SKU001\"]}"},
```

改造后：LiteFlow EL 表达式（存入数据库，支持热更新）

```text
-- 简单 AND 逻辑（等价于现有的 RuleExecutor.match）
THEN(
```

```text
-- 条件分支：不同店铺走不同审核链路
IF(scopeNode, THEN(productNode, tagNode)).ELSE(amountNode);

-- 并行校验：多个独立验证器同时执行，提升性能
WHEN(productNode, tagNode, amountNode);

-- 组合编排：先并行校验商品和标签，再串行校验金额
THEN(WHEN(productNode, tagNode), amountNode);

-- 子链路复用：把通用校验封装为子链路，多条规则共享
THEN(commonCheckChain, specificCheckNode);
```

### 12.6 AuditEngine 简化

改造前

```text
package com.yh.app.rsp.ordercenter.service.ecorder.engine;
import cn.hutool.core.date.DateUtil;
import cn.hutool.core.util.StrUtil;
import cn.hutool.json.JSONUtil;
import com.yh.app.rsp.enums.ecorder.AuditResultEnum;
import com.yh.app.rsp.ordercenter.dao.EcAuditLogMapper;
import com.yh.app.rsp.ordercenter.dao.EcAuditRuleMapper;
import com.yh.app.rsp.ordercenter.dto.audit.DelayAuditMessage;
import com.yh.app.rsp.ordercenter.dto.audit.RuleConfig;
import com.yh.app.rsp.ordercenter.dto.audit.ValidatorResult;
import com.yh.app.rsp.ordercenter.entity.EcAuditLogDO;
import com.yh.app.rsp.ordercenter.entity.EcAuditRuleDO;
import com.yh.app.rsp.ordercenter.executor.RuleExecutor;
import com.yh.app.rsp.ordercenter.service.ecorder.AuditLogAsyncService;
import com.yh.app.rsp.ordercenter.validator.OrderRuleValidateDTO;
import com.yh.app.rsp.ordercenter.validator.impl.PaymentTimeValidator;
import com.yh.app.rsp.producer.DelayAuditMessageProducer;
import lombok.extern.slf4j.Slf4j;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

import javax.annotation.Resource;
import java.util.Date;
import java.util.List;
import java.util.Objects;

/**
 * 审单引擎
 * 负责调用执行器并记录审核日志
 * 支持 RocketMQ 延迟消息机制实现支付后自动重审功能
 *
 * @author YCR
 */
@Slf4j
@Service
public class AuditEngine {

    @Resource
    private EcAuditRuleMapper ruleMapper;

    @Resource
    private EcAuditLogMapper logMapper;

    @Resource
    private PaymentTimeValidator paymentTimeValidator;

    @Resource
    private RuleExecutor ruleExecutor;

    @Resource
    private DelayAuditMessageProducer delayAuditMessageProducer;

    @Resource
    private AuditLogAsyncService auditLogAsyncService;

    /**
     * 审核订单
     *
     * @param order 订单DTO
     * @return 审核结果枚举
     */
    @Transactional(rollbackFor = Exception.class)
    public AuditResultEnum auditOrder(OrderRuleValidateDTO order) {
        // 1. 参数校验
        if (order == null || StrUtil.isBlank(order.getOrderId())) {
            log.warn("订单信息为空，跳过审核");
            return AuditResultEnum.PASS;
        }

        String orderId = order.getOrderId();

        // 2. 根据渠道店铺加载所有启用的规则(按 priority 倒序)
        // 通过适用店铺编码和适用渠道编码两个条件查询对应的规则
        List<EcAuditRuleDO> activeRules = ruleMapper.selectActiveRulesByShopAndChannel(order.getShopCode(), order.getChannelCode());

        if (activeRules == null || activeRules.isEmpty()) {
            return AuditResultEnum.PASS;
        }

        EcAuditRuleDO matchedRule = null;
        String matchedConfig = null;

        // 3. 遍历执行规则校验
        for (EcAuditRuleDO rule : activeRules) {
            try {
                String configStr = rule.getConfigJson();
                if (StrUtil.isBlank(configStr)) {
                    log.warn("规则{}配置为空，跳过该规则", rule.getId());
                    continue;
                }

                List<RuleConfig> configList = JSONUtil.toList(configStr, RuleConfig.class);
                if (configList == null || configList.isEmpty()) {
                    log.warn("规则{}配置解析为空，跳过该规则", rule.getId());
                    continue;
                }

                // 执行完整校验
                if (ruleExecutor.match(order, configList)) {
                    matchedRule = rule;
                    matchedConfig = configStr;
                    log.info("订单{}命中规则[{}]，规则ID：{}", orderId, rule.getRuleName(), rule.getId());
                    // 只要命中一条，立刻停止返回审核不通过
                    handleReject(order, matchedRule, matchedConfig);
                    return AuditResultEnum.REJECT;
                }
            } catch (Exception e) {
                log.error("规则解析执行异常, ruleId={}, ruleName={}, orderId={}", rule.getId(), rule.getRuleName(), orderId, e);
                // 异常情况下跳过当前规则，继续下一条
            }
        }

        // 4. 如果未命中任何规则，检查是否需要延迟审核
        // 延迟审核配置只会有0条或1条，找到第一个配置了延迟时间的规则即可
        EcAuditRuleDO delayRule = null;
        Integer delayMinutes = null;
        Boolean strictMode = null;

        for (EcAuditRuleDO rule : activeRules) {
            try {
                delayMinutes = rule.getDelayMinutes();
                strictMode = rule.getStrictMode();
                // 如果找到配置了延迟时间的规则，立即停止查找
                if (delayMinutes != null && delayMinutes > 0) {
                    delayRule = rule;
                    break;
                }
            } catch (Exception e) {
                log.error("延迟审核配置解析异常, ruleId={}, ruleName={}, orderId={}", rule.getId(), rule.getRuleName(), orderId, e);
                // 异常情况下跳过当前规则，继续查找
            }
        }

        // 如果找到延迟审核规则，执行支付时间验证
        if (delayRule != null) {
            try {
                ValidatorResult result = paymentTimeValidator.validateWithResult(order, delayMinutes, strictMode);
                if (!result.isSuccess()) {
                    // 支付时间不足，需要延迟审核
                    log.info("订单{}未命中规则但支付时间不足，需要延迟审核，规则ID：{}，延迟分钟数：{}", orderId, delayRule.getId(), delayMinutes);
                    handleDelay(order, delayRule, delayRule.getConfigJson(), result);
                    return AuditResultEnum.PENDING;
                }
            } catch (Exception e) {
                log.error("延迟审核验证异常, ruleId={}, ruleName={}, orderId={}", delayRule.getId(), delayRule.getRuleName(), orderId, e);
                // 验证异常，继续执行后续逻辑
            }
        }

        // 5. 未命中任何规则且不需要延迟审核 -> 通过
        return AuditResultEnum.PASS;
    }

    /**
     * 处理审核通过
     *
     * @param order  订单DTO
     * @param rule   命中的规则
     * @param config 规则配置
     */
    @Transactional(rollbackFor = Exception.class)
    public void handlePass(OrderRuleValidateDTO order, EcAuditRuleDO rule, String config) {
        // 1. 记录日志 (PASS) - 异步入库
        EcAuditLogDO auditLog = new EcAuditLogDO();
        auditLog.setOrderId(order.getOrderId());
        auditLog.setRuleId(Objects.requireNonNull(rule).getId());
        auditLog.setRuleName(rule.getRuleName());
        auditLog.setAuditResult(AuditResultEnum.PASS.getCode());
        auditLog.setDetailMsg("命中规则[" + rule.getRuleName() + "]，自动审核通过");

        // 核心：保存快照，将当时的配置 JSON 存入日志
        auditLog.setSnapshotJson(config);
        // 系统自动
        auditLog.setCreator(0L);
        // 异步入库
        auditLogAsyncService.insertAuditLogAsync(auditLog);

        // 2. 执行订单流转业务
        log.info("订单 {} 自动审核通过，规则ID：{}", order.getOrderId(), rule.getId());
    }

    @Transactional(rollbackFor = Exception.class)
    public void handleReject(OrderRuleValidateDTO order, EcAuditRuleDO rule, String config) {
        // 1. 记录日志 (REJECT) - 异步入库
        EcAuditLogDO auditLog = new EcAuditLogDO();
        auditLog.setOrderId(order.getOrderId());
        auditLog.setAuditResult(AuditResultEnum.REJECT.getCode());
        auditLog.setDetailMsg("自动审核不通过!");
        auditLog.setRuleId(rule.getId());
        auditLog.setRuleName(rule.getRuleName());
        // 命中规则快照
        auditLog.setSnapshotJson(config);
        // 系统自动
        auditLog.setCreator(0L);
        // 异步入库
        auditLogAsyncService.insertAuditLogAsync(auditLog);

        // 2. 保持订单状态不变，或推送到人工审核队列
        log.info("订单 {} 进入人工审核", order.getOrderId());

    }

    /**
     * 处理延迟审核
     * 记录延迟审核日志并发送 RocketMQ 延迟消息
     *
     * @param order       订单DTO
     * @param rule        命中的规则
     * @param config      规则配置
     * @param delayResult 延迟验证结果
     */
    @Transactional(rollbackFor = Exception.class)
    public void handleDelay(OrderRuleValidateDTO order, EcAuditRuleDO rule, String config, ValidatorResult delayResult) {
        // 1. 记录延迟审核日志 (DELAY)
        EcAuditLogDO auditLog = new EcAuditLogDO();
        auditLog.setOrderId(order.getOrderId());
        auditLog.setRuleId(Objects.requireNonNull(rule).getId());
        auditLog.setRuleName(rule.getRuleName());
        auditLog.setAuditResult(AuditResultEnum.PENDING.getCode());
        auditLog.setDetailMsg(String.format("命中规则[%s]但支付时间不足%分钟，已支付%d分钟，延迟审核", rule.getRuleName(), delayResult.getDelayMinutes(), delayResult.getElapsedMinutes()));

        // 保存快照
        auditLog.setSnapshotJson(config);

        // 设置延迟审核相关字段
        auditLog.setDelayMinutes(delayResult.getDelayMinutes());
        Date expectedTime = DateUtil.offsetMinute(new Date(), delayResult.getDelayMinutes());
        auditLog.setDelayExpectedTime(expectedTime);
        auditLog.setDelayStatus(AuditResultEnum.PENDING.getCode());

        // 系统自动
        auditLog.setCreator(0L);
        // 同步插入以获取ID（用于发送延迟消息）
        logMapper.insert(auditLog);

        // 2. 发送 RocketMQ 延迟消息
        try {
            DelayAuditMessage message = new DelayAuditMessage();
            message.setOrderId(order.getOrderId());
            message.setShopCode(order.getShopCode());
            message.setChannelCode(order.getChannelCode());
            message.setOrderType(order.getOrderType());
            message.setPaymentTime(order.getPaymentTime() != null ? order.getPaymentTime().getTime() : null);
            message.setDelayMinutes(delayResult.getDelayMinutes());
            message.setStrictMode(rule.getStrictMode());
            message.setRuleId(rule.getId());
            message.setRuleName(rule.getRuleName());
            message.setAuditLogId(auditLog.getId());
            message.setRetryCount(0);
            message.setMaxRetryCount(3);
            message.setCreateTime(System.currentTimeMillis());

            delayAuditMessageProducer.sendDelayMessage(message);

            log.info("订单{}延迟审核消息发送成功，规则ID：{}，预计执行时间：{}", order.getOrderId(), rule.getId(), expectedTime);
        } catch (Exception e) {
            log.error("订单{}延迟审核消息发送失败", order.getOrderId(), e);
            // 消息发送失败，异步更新日志状态
            auditLog.setDelayStatus("FAILED");
            auditLogAsyncService.updateAuditLogAsync(auditLog);
        }
    }
}
```

改造后的引擎核心逻辑大幅简化，规则遍历、验证器调度全部交给 LiteFlow：

```text
@Service
public class AuditEngine {

    @Resource
    private FlowExecutor flowExecutor;

    @Resource
    private AuditLogAsyncService auditLogAsyncService;

    @Resource
    private DelayAuditMessageProducer delayAuditMessageProducer;

    public AuditResultEnum auditOrder(OrderRuleValidateDTO order) {

            return AuditResultEnum.PASS;

        AuditContext ctx = new AuditContext();
        ctx.setOrder(order);

        LiteflowResponse response = flowExecutor.execute2Resp("auditChain", null, ctx);

            return ctx.getResult();
        }

        // 异常兜底：放行
        log.error("审单引擎执行异常, orderId={}", order.getOrderId(), response.getCause());
```

### 12.7 改造后的能力对比

| 能力           | 自研引擎（当前）         | LiteFlow 改造后                        |
| -------------- | ------------------------ | -------------------------------------- |
| 验证器组合逻辑 | 只支持 AND               | AND / OR / NOT / IF-ELIF-ELSE 任意组合 |
| 规则热更新     | 改 DB 后需重启或手动刷新 | liteflow-rule-sql 自动轮询热加载       |
| 执行可视化     | 只有散落的日志           | LiteFlow 自带执行步骤链路追踪          |
| 验证器并行执行 | 不支持                   | WHEN(a, b, c) 并行执行，提升性能       |
| 规则表达能力   | JSON 数组，只能线性 AND  | EL 表达式，支持条件分支、循环、子流程  |
| 节点回滚       | 无                       | 节点支持 rollback() 方法               |
| 子链路复用     | 不支持                   | 通用校验封装为子链路，多规则共享       |

### 12.8 需要保留不变的部分

并非所有组件都需要改造，以下部分应保持现有设计：

- 延迟消息机制（RocketMQ Producer / Consumer）—— 定时调度不属于规则引擎范畴，LiteFlow 不处理"未来某个时刻触发"的场景

- 审核日志 + 快照（AuditLogAsyncService）—— 可以封装为 LiteFlow 的后置节点（FinallyComponent），链路结束后自动记录

- ValidatorResult / DelayAuditMessage —— 数据模型不变，通过 AuditContext 在节点间传递

- OrderRuleValidateConverter —— 领域模型转换依然在引擎外部完成

```text
Phase 2 ─ 接入数据库规则源
  ├── 引入 liteflow-rule-sql
  ├── ec_audit_rule 表新增 el_expression 字段（替代 config_json）
  ├── 配置 LiteFlow 从数据库加载规则，实现热更新
  └── 管理后台适配新的规则配置方式

Phase 3 ─ 利用 LiteFlow 增强能力
  ├── 将审核日志封装为 FinallyComponent，自动记录链路快照
  ├── 引入 WHEN 并行编排，提升多验证器场景的性能
  ├── 利用 IF-ELIF-ELSE 实现复杂条件分支
  ├── 抽取通用校验为子链路，多规则复用
  └── 逐步淘汰 RuleExecutor，由 LiteFlow 完全接管编排
```

Phase 1 改动量最小，核心目的是验证 LiteFlow 与现有架构的兼容性。由于现有验证器本身就是无状态、单职责的，校验逻辑可以几乎原封不动地搬入 LiteFlow 节点——这也从侧面验证了最初策略模式设计的正确性：好的抽象不仅服务于当下，也为未来的架构升级铺好了路。
