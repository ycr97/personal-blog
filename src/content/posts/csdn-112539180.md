### 一丶线程创建
Java创建线程的方式主要是有继承Thread类和实现Runable接口年重要方式还有使用线程池的方式实际上也是上面两种方式的变种
#### 方式一:继承Thread类

 1. 创建一个继承于Thread类的子类
 2. 重写run方法 --> 此线程需要做的事写在run方法中
 3. 创建Thread类之类的对象
 4. 通过此对象调用start方法

```java
/**
 * @author ycr
 * @date 2021/1/12
 */
public class ThreadDemo {

    public static void main(String[] args) {
        Demo demo = new Demo();
        demo.setName("demo");
        demo.start();
        for (int i = 0; i < 100; i++) {
            if (i % 2 == 0) {
                System.out.println(Thread.currentThread().getName() + ":" + i);
            }
        }

    }
}

class Demo extends Thread {

    @Override
    public void run() {
        for (int i = 0; i < 100; i++) {
            if (i % 2 == 0) {
                System.out.println(Thread.currentThread().getName() + ":" + i);
            }
        }
    }
}
```
#### Thread类中常用方法

 1. start(): 启动当前线程,调用当前线程的run()方法
 2. run():通常要重写Thread类中的此方法,将创建的线程要执行的操作写在此方法中
 3. currentThread():静态方法,返回执行当前代码的线程
 4. getName():获取当前线程的名字
 5. setName():设置当前线程名字
 6. yield():释放当前CPU的执行权
 7. join():在线程A中调用线程B的join()方法,此时线程A会进入阻塞状态,直到线程B完全执行完之后,线程A才结束阻塞状态
 8. ~~stop()~~ :已过时,不推荐使用,强制结束当前线程
 9. sleep(long millis): 让当前线程"睡眠"指定的毫秒数,在指定的毫秒数内当前线程为阻塞状态
 10.isAlive():判断当前线程是否为存活状态 
#### 线程的优先级
 1. MIN_PRIORITY : 1
NORM_PRIORITY : 5 默认优先级
MAX_PRIORITY : 10
 2. 如何获取和设置线程优先级:
	getPriority():获取线程优先级
	setPriority(int P):设置线程优先级

**说明**:高优先级的线程要抢占低优先级线程的cpu执行权.但是只是从概率上讲,高优先级的线程高概率被执行,并不意味着只有当高优先级线程执行完低优先级才会被执行.
#### 方式二:实现Runable接口

 1. 创建一个实现了Runable接口的类
 2. 实现类中实现Runable接口中的抽象方法:run()
 3. 创建实现类的对象
 4. 将此类对象作为参数传递到Thread类的构造器中,创建Thread类的对象
 5. 通过Thread类对象的start方法启动线程

#### 比较两种创建线程的方式
开发中优先选择实现Runable接口的方式
**原因 :**

 1. 实现的方式没有单继承的局限性
 2. 实现的方式更适合来处理多个线程有共享数据的情况

**联系:** Thread也实现了Runable接口
**相同:** 两种方式都需要重写run()方法,将要创建线程的执行的逻辑放于此方法中

#### 线程的生命周期
JDK中用Thread.Status定义了线程的几种状态

 - **新建:** 当Thread类及其子类被声明并创建时,新生的线程对象处于新建状态
 - **就绪:** 处于新建状态的线程调用了start()方法,将进入线程队列等待cpu时间片,此时该线程已具备运行条件,只是未获得cpu时间片.
 - **运行:** 当就绪的线程被调度获得cpu资源时,便进入运行状态,run()方法定义了线程操作和功能
 - **阻塞:** 在某种特殊情况下,被人为挂起或执行输入输出操作时,让出cpu并临时中止自己的执行,进入阻塞状态
 - **死亡:** 线程完成了它全部的工作或线程提前被强制性地中止或出现异常导致结束

#### 线程安全
在Java中,我们通过同步机制来解决线程的安全问题.操作同步代码时只能有一个线程参与,此时相当于单线程过程此时效率较低

##### 方式一: synchronized 关键字
```java
synchronized (同步监视器)  {
    //需要被同步的代码块
} 
```
**说明:** 
 1. 操作共享数据的代码,即为被需要被同步的代码
 2. 共享数据:多个线程共同操作的变量
 3. 同步监视器,俗称"锁".任何一个类的对象都可以充当锁
 		**要求:** 多个线程必须要共用同一把锁
 	**补充:** 在实现Runable接口方式创建线程的时候可使用对象本身(this)作为同步监视器

在使用继承Thread类的情况下慎用this作为同步监视器,可以考虑当前类来充当同步监视器

**同步方法:**

如果操作共享数据的代码完整的声明在一个方法中,那么我们不妨将此方法声明同步的,同步监视器为this
**总结:** 

 1. 同步方法依然会涉及到同步监视器,只是不需要我们显式的声明
 2. 非静态方法的同步监视器是自己:this
静态方法的的同步监视器是:当前类本身

#### 单例模式懒汉式线程安全

```java
class Bank {

    private static Bank bank = null;

    /**
     * 此时为线程不安全
     *
     * @return instance
     */
    public static Bank getInstance() {
        if (bank == null) {
            bank = new Bank();
        }
        return bank;
    }

    /**
     * 使用同步锁线程安全,但是此写法效率不高不推荐
     *
     * @return instance
     */
    public static synchronized Bank getInstance1() {
        if (bank == null) {
            bank = new Bank();
        }
        return bank;
    }

    /**
     * 同上写法
     *
     * @return instance
     */
    public static Bank getInstance2() {
        synchronized (Bank.class) {
            if (bank == null) {
                bank = new Bank();
            }
            return bank;
        }
    }

    /**
     * 此方法效率比上面效率更高
     *
     * @return instance
     */
    public static Bank getInstance3() {
        if (bank == null) {
            synchronized (Bank.class) {
                if (bank == null) {
                    bank = new Bank();
                }
            }
        }
        return bank;
    }

    private Bank() {

    }
}
```

#### 线程的死锁问题
下面代码会导致线程死锁

```java
/**
 1. @author ycr
 2. @date 2021/1/15
 */
public class DeadlockDemo {

    public static void main(String[] args) {

        StringBuilder s1 = new StringBuilder();
        StringBuilder s2 = new StringBuilder();

        new Thread(() -> {
            synchronized (s1) {
                try {
                    Thread.sleep(100);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
                s1.append("a");
                s2.append("1");

                synchronized (s2) {
                    s1.append("b");
                    s2.append("2");
                    System.out.println(s1);
                    System.out.println(s2);
                }
            }
        }).start();

        new Thread(() -> {
            synchronized (s2) {
                try {
                    Thread.sleep(100);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
                s1.append("c");
                s2.append("3");

                synchronized (s1) {
                    s1.append("d");
                    s2.append("4");

                    System.out.println(s1);
                    System.out.println(s2);
                }
            }
        }).start();
    }
}
```
**线程死锁的理解:**不同的线程分别占用对方需要的同步资源不放弃,都在等待对方放弃自己需要的同步资源,就形成了现成的死锁
**说明:**
 3. 出现死锁后不会出现异常,不会出现提示,只是所有的线程都处于阻塞状态,无法继续执行.
 4. 在使用线程的同步的时候应避免死锁问题.
 
**解决办法:**

 1. 专门的算法,原则
 2. 尽量减少同步资源的定义
 3. 尽量避免嵌套同步

#### 方式二:Lock锁
从JDK1.5开始Java提供java.util.concurrent.locks.Lock接口来控制多个线程对共享资源访问的控制,锁提供了对共享资源的独占访问每次只能一个对象对Lock对象加锁,线程在访问共享资源之前应先获得Lock对象.

```java
public class LockDemo {

    public static void main(String[] args) {
        WindowTest test = new WindowTest();

        Thread t1 = new Thread(test, "T1");
        Thread t2 = new Thread(test, "T2");
        Thread t3 = new Thread(test, "T3");

        t1.start();
        t2.start();
        t3.start();
    }
}

class WindowTest implements Runnable {

    // 1.实例化Lock
    private Lock lock = new ReentrantLock();

    private int ticket = 100;

    @Override
    public void run() {
        while (true) {
            // 调用锁定的方法
            lock.lock();
            try {
                if (ticket > 0) {
                    try {
                        Thread.sleep(200);
                        System.out.println(Thread.currentThread().getName() + ": " + ticket);
                        ticket--;
                    } catch (InterruptedException e) {
                        e.printStackTrace();
                    }
                } else {
                    break;
                }
            } finally {
                // 调用解锁的方法
                lock.unlock();
            }
        }
    }
}
```
**面试题:** Lock和 synchronized异同
相同: 二者都可以用来解决线程安全问题
不同:synchronized机制在执行完相应同步代码后,自动释放同步监视器
		lock需要手动的启动同步监视器(lock()),同时结束同步也需要手动的实现(unlock())
		使用Lock锁,JVM将花费更少的时间来调度线程,性能更好.并且具有更好的扩展性(提供更多的子类)
		**使用优先级顺序:**
		Lock > 同步代码快(已经进入了方法体,分配了相应资源) > 同步方法(在方法体之外)
